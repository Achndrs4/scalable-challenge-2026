## Installation
Instructions on installation can be found in [Readme.md](Readme.md). This section covers design decisions and differences between this pipeline and a fully productionised setup.

## File Validation
The first step of this pipeline is ingesting our JSONL file with records. We do this by splitting the read into different process pools and validating multiple lines simultaneously, until we have a filtered output file in terms of format (though not in terms of the actual data it is meant to represent). This should be done here, and not by dbt, because the cleanup here does not involve business logic — simply validation that the file can be parsed, and if not, at least partially parsed.

## DB Ingestion

The raw ingestion layer (dbt) adds a microsecond-resolution `batch_timestamp` and a unique `batch_id` (generated by dbt itself) to every row. This means that if a corrected version of a file is re-ingested, the staging layer can identify it as a newer batch and overwrite the older values via the `delete+insert` strategy.

While the previous raw layer gave us a valid raw file with all of the "transactions" we need, we now need to apply some filtering and splitting that can be used across the semantic layer. Our staging table filters for meaningless requests (like those without a user_name) and de-duplicates our rows. The unique key — user_name, recording_msid, and listened_at — uses a delete+insert strategy so if a file comes in with bad values (for example, an upstream service corrupted all artist titles), we can re-ingest it and the corrected values will overwrite the old ones.

## Business Logic / Semantic Layer
Now, with this table, we can create a table dedicated for business queries. Having it on a separate table guarantees the I/O does not overlap with either of the above two flows. The table was chosen to interact neatly with the saved queries.
